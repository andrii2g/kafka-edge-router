# Kafka Edge Router v0.1.0-rc.1

This is the first production-candidate release. It establishes the public protocol,
security, bounded-resource, observability, deployment, and release-signing baselines for
operator evaluation. It is a release candidate, not a declaration that every workload
will meet the included benchmark results.

## Delivery semantics

- Kafka ingestion is at least once. Records can be duplicated around crashes, asynchronous
  commits, rebalances, and operator replay.
- WebSocket, SSE, and gRPC delivery is best effort after bounded local queue acceptance.
  A process crash or disconnect can lose messages that Kafka has already committed.
- Live clients must retain and deduplicate `message_id`; there is no durable client cursor,
  acknowledgement store, or replay window.
- Webhook delivery is explicitly `volatile` or Kafka-backed `durable`. Durable mode retains
  retry state across restarts but can still repeat a callback after remote success and a
  crash before commit. Receivers must persist the idempotency key.
- Ordering is Kafka partition-local and depends on stable entity keys. No global order is
  promised across partitions, nodes, destinations, or reconnects.
- HTTP and gRPC publishing acknowledge Kafka coordinates, not downstream processing.

## Security and operational limits

- Production mode requires loopback-only daemon listeners behind a TLS proxy and
  cryptographic JWT/JWKS or proxy-mTLS identity.
- Tenant identity is never wildcardable. Subscription filters are rewritten to the
  authenticated tenant.
- Every connection, subscription set, command rate, publish rate, payload, queue, retry,
  JWKS file, DNS result set, and load-generator run has a configured hard bound.
- Webhook redirects and ambient egress proxies are disabled. DNS answers are revalidated
  and pinned for each attempt; private and special addresses are rejected by default.
- Full-stream topology requires one unique Kafka consumer group per router pod. Kafka read
  and matching work therefore scales with router replica count.
- The Kubernetes manifests require operator-created configuration, JWKS, and TLS Secrets;
  they are not embedded in the image or release assets.

## Known limits

- No exactly-once or guaranteed end-to-end live delivery.
- No persisted dynamic subscriptions or missed-event replay for live protocols.
- No shared-group peer forwarding; every node consumes the complete stream.
- No arbitrary payload-expression matching; routing uses bounded Kafka headers only.
- Route mutations use one process-wide mutex. Dispatch remains lock-independent, but very
  high subscription churn must be capacity-tested against the published benchmark method.
- Standard Kubernetes NetworkPolicy cannot enforce DNS-name webhook allowlists. Application
  SSRF validation remains mandatory, and clusters should add an egress gateway where FQDN
  policy is required.
- Performance evidence applies only to the recorded hardware, configuration, payload,
  fan-out, and commit. Operators must reproduce the scenarios for their own capacity plan.

## Artifact verification

Release assets include Linux amd64/arm64 binaries, `SHA256SUMS`, an SPDX JSON SBOM,
keyless Cosign bundles, and GitHub build-provenance attestations. The GHCR manifest is
multi-architecture, scanned under the repository HIGH/CRITICAL policy, signed by its
immutable digest, and runs as UID/GID `10001:10001`.
